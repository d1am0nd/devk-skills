# Selecting diffs — what to read, what to skim, what to skip

Goal: form an intent hypothesis from the *minimum* number of full file diffs. The main agent's context budget is limited; this is where you spend it.

## Bucketing each file

For each changed file, place it into one of four buckets.

### Read full diff

- Application code (commonly under `src/`, `lib/`, `app/`, `internal/`, `pkg/`, `cmd/`) with a moderate change size — say, under ~300 changed lines.
- Configuration changes that look meaningful (feature flags, route registration, dependency injection wiring).
- Documentation that explains the change itself (a focused CHANGELOG entry, a README section about what was added).
- Anything where the diff itself reveals intent — function bodies, route handlers, schema/migration changes, new modules.

### Read partial diff

- Application files with >300 changed lines: read the diff in hunks. Skim past pure formatting/whitespace runs and grouped trivia.
- Files with both meaningful and trivial changes mixed (a refactor that also reformats large blocks).
- Use `git diff <range> -- <file>` and read it in chunks if you have to. Don't read the whole file's post-change contents in the main agent — that's a step-6 subagent's job.

### Grep, don't read

- **Test files, especially large ones.** Step 1's job is intent — test bodies rarely reveal it better than the code they test. Run a quick `grep` for test names/descriptions:
  - JS/TS: `grep -E "(describe|it|test)\(" <file>`
  - Python: `grep -E "def test_" <file>`
  - Go: `grep "func Test" <file>`
  - Ruby: `grep -E "(describe|it)\s" <file>`
- Localization / translation files (`*.po`, `*.json` under `locales/`, `i18n/`) — grep for changed keys; don't read the values.
- API spec files (OpenAPI, proto, GraphQL schema) — grep for changed endpoint/message/type names.

### Skip entirely

- **Lockfiles**: `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`, `Cargo.lock`, `Pipfile.lock`, `poetry.lock`, `go.sum`, `Gemfile.lock`. Note presence ("dependencies updated") and move on. If a lockfile changed but no manifest (`package.json`, `Cargo.toml`, etc.) did, that itself is worth a question to the human.
- **Generated code**: anything with a `// generated by` / `# generated` / `# Code generated` marker. Files in `dist/`, `build/`, `node_modules/`, `vendor/`, `__generated__/`, `*.pb.go`, `*.pb.ts`, `*.gen.ts`, `schema.gen.*`, `openapi/generated/`, etc.
- **Snapshot fixtures**: `__snapshots__/`, `*.snap`, `*.snapshot`. Their content is downstream of code; review the code instead.
- **Binary or near-binary diffs**: images, fonts, compiled assets, minified bundles.

## Tests in the intent phase

Tests usually don't reveal intent better than the code they test. Use the grep approach to confirm tests exist and exercise the new behavior, then move on. Two exceptions:

- The *only* changed files are test files. Then yes, read them — the change IS test work.
- A test name itself is genuinely surprising given the rest of the diff (e.g. `test_user_can_skip_2fa` on a security-touching change). Read that one test's body.

## A useful order

1. Look at commit subjects from `git log --oneline <range>` first — they often state intent directly.
2. Read short, focused application code: schema/model changes, new functions added, new modules.
3. Then look at any updated configuration / wiring (routes, registrations, feature flags).
4. Then peek at test names to confirm the change is exercised.
5. Read documentation files (CHANGELOG, README updates) early if present — they often contain the cleanest intent statement.

## When you have enough

Stop reading when you can write a 1–2 sentence intent statement that explains *what the change does* and *why* with reasonable confidence. Beyond that, more reading is wasted context — questions to the human in step 5 are cheaper.

## Track suspicions as you read

Spotted something fishy? Don't review yet — log it in a small structured list:

```
- src/api/auth.ts:42 — exception swallow with no logging
- internal/db/migration.sql — adds column, didn't see code that uses it
- pkg/router/route.go:117 — signature change, no caller updates in the diff
```

This list has two downstream uses:

1. The unresolved entries become candidate follow-ups for the human in step 5.
2. The still-unresolved-after-step-5 entries get handed to the relevant section reviewer in step 6 — framed as *"investigate or rule out"*. The point: subagents shouldn't have to re-derive what you already noticed in step 3.

Common things worth logging:

- A removed test with no obvious replacement.
- A new `TODO` / `FIXME` committed.
- An exception handler that swallows errors silently.
- A condition that looks flipped or backwards.
- A new dependency added without an obvious use site in the same diff.
- A schema/migration change without corresponding code touching the new column/field.
- A function signature changed in one place without obvious caller updates in the same diff.
- Authentication / authorization / crypto code touched.
