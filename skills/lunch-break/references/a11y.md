# Lens: Accessibility (a11y)

## Scope

Audit the project for accessibility issues using WCAG 2.1 AA as the reference standard. Cover semantic HTML, keyboard navigation, screen reader experience, color contrast, focus management, and form accessibility.

This lens cares about:
- Semantic HTML (proper landmarks, headings, lists, buttons vs. divs)
- Keyboard-only navigation (every interactive thing reachable and operable)
- Focus management (visible focus, logical order, focus trapping in modals)
- Screen reader experience (labels, ARIA used correctly, alt text)
- Color contrast (text/background, interactive states)
- Form accessibility (labels, error association, fieldsets where appropriate)
- Motion / reduced motion preferences
- Skip links and bypass mechanisms
- Touch target sizes (mobile)
- Language attributes
- Live regions and dynamic content announcements

This lens does NOT cover:
- General visual design — `ui` lens
- Copy quality — `copy` lens

## Phase 1 — Scout

Dispatch 3–4 subagents:

1. "Identify the framework and any a11y libraries already in use (e.g. Radix, Reach UI, react-aria, headless UI). Report what accessibility infrastructure exists in this codebase."
2. "List all interactive component types in this app (buttons, links, custom controls, dialogs, menus, accordions, tabs, comboboxes, tooltips, etc.). Note which are using native elements vs. custom-built."
3. "List all forms in the app and the field types used in each."
4. "Check for any existing a11y testing in the project (axe, jest-axe, lighthouse CI, manual test docs, axe DevTools usage notes). Report what exists."

## Phase 2 — Deep dive

Dispatch 8–15 subagents (liberal) or 20+ across waves (maximum):

- "Audit semantic HTML across the app. Find every `<div>` or `<span>` that's being used as an interactive element (click handlers without role/keyboard support). Flag and propose semantic replacements (usually: use a `<button>`)."
- "Trace keyboard navigation through every key flow (login, primary task, settings, checkout if applicable). Identify focus traps, skipped elements, illogical tab order."
- "Check focus visibility everywhere. Is there a visible focus indicator on every interactive element? Any `outline: none` without a replacement focus style?"
- "Audit every modal / dialog / popover. Does focus move into it on open? Does it return to the trigger on close? Is focus trapped while open? Can it be closed with Escape? Is there an accessible name?"
- "Audit every form. Does every input have a programmatically associated label? Are errors associated via `aria-describedby` and announced? Are required fields marked accessibly (not color-only)?"
- "Check ARIA usage. Find any wrong ARIA — `aria-label` on a non-interactive div, `role` conflicting with native semantics, `aria-hidden` on focusable elements, missing `aria-expanded` on disclosure widgets."
- "Audit images. Decorative images should have `alt=''`; informative images should have descriptive alt. Flag missing, redundant, or filename-as-alt cases."
- "Check color contrast on all text and interactive states (default, hover, focus, disabled, error). Use WCAG AA: 4.5:1 for normal text, 3:1 for large text and UI components. List failures with measured ratio."
- "Audit custom widgets (tabs, accordions, comboboxes, menus, tree views). Compare to ARIA Authoring Practices. Flag deviations."
- "Check for `prefers-reduced-motion` handling. Are animations and transitions respecting it?"
- "Check landmark structure on each page. Is there exactly one `<main>`? Are `<nav>` / `<header>` / `<footer>` landmarks set?"
- "Check page language attributes (`<html lang>`) and any sections with different languages (need `lang` on those elements)."
- "Check touch target sizes on mobile (minimum 44×44px per WCAG). Flag undersized targets, especially close-together ones."
- "Audit dynamic content updates — toasts, live updates, search-as-you-type. Are they announced via live regions? Is the right `aria-live` politeness used?"
- "Look for skip links. Is there a 'skip to main content' link near the top of the page? Does it work?"

For **maximum** mode, second wave: take the most-affected component or flow and dispatch subagents to do a deeper end-to-end pass — for example, "Take the checkout flow, trace it with keyboard only, and trace it with VoiceOver mental-model emulation. Report every friction point."

## Finding shape

For each finding:
- **Cite the WCAG criterion** when relevant (e.g. "1.4.3 Contrast (Minimum)", "2.1.1 Keyboard", "4.1.2 Name, Role, Value").
- **Quote the specific element/file** at fault.
- **Describe how to verify the issue** in human terms (e.g. "Tab to the search bar — focus indicator is invisible").
- **Propose a concrete fix** at the code level.
- **Severity:** blocker (blocks a class of users), serious (bad experience), moderate (papercut).
